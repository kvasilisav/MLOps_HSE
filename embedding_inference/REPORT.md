# Отчёт: оптимизация inference pipeline (rubert-mini-frida)

## Выбор метрик

Использованы метрики, релевантные для HTTP-сервиса эмбеддингов:

- **Latency (задержка)** — время от запроса до ответа. Считаем mean и p95 (и для части 1 — p50, p99), чтобы оценить типичное и «худшее» поведение. Важно для интерактивных сценариев.
- **Throughput (пропускная способность)** — число успешных запросов в секунду. Показывает, сколько нагрузки выдерживает сервис при длительном прогоне (части 1 и 2) или при параллельных клиентах (часть 3).
- **Потребление ресурсов** замерялось через `docker stats` во время бенчмарка (CPU %, RAM).

Все замеры выполнены на **CPU**.

---

## Подходы

| Часть | Описание |
|-------|----------|
| **1** | Базовый инференс: загрузка модели с HuggingFace (SentenceTransformer), один запрос — один forward, FastAPI. |
| **2** | Модель в ONNX (export через optimum), инференс через onnxruntime, один запрос — один вызов сессии. |
| **3** | Та же ONNX-модель + динамическое батчирование: очередь запросов, воркер собирает батч и обрабатывает его за один forward, ответы раздаются по futures. |

## Анализ ускорения

**Part 2 vs Part 1 (переход на ONNX):**

| Метрика | Part 1 | Part 2 | Ускорение (Part 2 / Part 1) |
|---------|--------|--------|-----------------------------|
| Latency (mean) | 20.96 ms | 6.37 ms | **3.29×** быстрее |
| Throughput | 57.65 req/s | 166.49 req/s | **2.89×** выше |

Переход с HuggingFace/PyTorch на ONNX даёт примерно **3.3×** по задержке и **~2.9×** по пропускной способности за счёт оптимизированного рантайма onnxruntime и фиксированного графа без накладных расходов фреймворка.

**Part 3 vs Part 2 (динамическое батчирование):**

| Метрика | Part 2 | Part 3 | Изменение (Part 3 / Part 2) |
|---------|--------|--------|-----------------------------|
| Latency (mean) | 6.37 ms | 5.67 ms | **1.12×** быстрее (≈11% меньше задержка) |
| Throughput | 166.49 req/s | 242.85 req/s | **1.46×** выше |

Батчирование объединяет несколько запросов в один forward, лучше загружает CPU при конкурентных клиентах и даёт прирост throughput **~1.5×** при слегка меньшей средней задержке.

**Part 3 vs Part 1 (суммарный эффект):**

| Метрика | Part 1 | Part 3 | Ускорение (Part 3 / Part 1) |
|---------|--------|--------|-----------------------------|
| Latency (mean) | 20.96 ms | 5.67 ms | **3.70×** быстрее |
| Throughput | 57.65 req/s | 242.85 req/s | **4.21×** выше |

От базового HF-варианта до ONNX с батчированием: задержка уменьшается в **~3.7 раза**, пропускная способность вырастает в **~4.2 раза**.

**Потребление ресурсов:** Part 1 использует меньше RAM (~326 MiB) за счёт одной модели в памяти; Part 2 и Part 3 держат ONNX-модель и токенайзер (~446–449 MiB). Загрузка CPU под нагрузкой сопоставима (170–185%) — все варианты упираются в ядра, при этом Part 2 и Part 3 при той же загрузке обрабатывают в разы больше запросов в секунду.

---

## Выводы

- **Part 1 → Part 2:** переход на ONNX даёт основное ускорение (≈3.3× по latency, ≈2.9× по throughput) при умеренном росте RAM.
- **Part 2 → Part 3:** батчирование добавляет ≈1.5× по throughput и немного снижает среднюю задержку при конкурентной нагрузке.
- **Part 1 → Part 3:** суммарно ≈3.7× по latency и ≈4.2× по throughput при сопоставимом CPU и умеренно большем объёме RAM.

Для продакшена при ограничении по CPU разумно использовать Part 2 или Part 3 в зависимости от профиля нагрузки: стабильный поток запросов — Part 3, редкие одиночные запросы — Part 2.
