# Model Registry — отчёт по ДЗ

## 1. Проблемы текущего состояния

**Что есть сейчас:** папка `models/` с вложенной структурой, куда команды складывают модели. Никакой метаинформации.

| Проблема | Описание |
|----------|----------|
| **Нет контекста** | Непонятно, что за модель, кто обучил, когда, на каких данных |
| **Дубликаты и версии** | Имена вроде `model_v1_v0` — версионирование в названии папки, легко запутаться |
| **Нет поиска** | Чтобы найти модель, нужно знать путь. Фильтрация по задаче/команде/дате невозможна |
| **Нет связи с экспериментом** | Не видно, какие гиперпараметры, метрики, датасет |
| **Риск перезаписи** | Два запуска с одинаковым именем — один затрет другой |
| **Нет lifecycle** | Нельзя пометить модель как production/staging/deprecated |

---

## 2. Требования

### Функциональные

- **Регистрация модели** — загрузка артефакта + метаданные (имя, команда, версия, теги)
- **Поиск и фильтрация** — по команде, тегам, дате, имени
- **Версионирование** — одна логическая модель может иметь несколько версий
- **Статусы** — staging / production / archived
- **Скачивание** — получение файлов модели по ID или версии

### Нефункциональные

- **Простота** — минимум зависимостей, быстрый старт
- **Надёжность** — файлы не теряются при сбоях
- **Масштабируемость** — возможность хранить тысячи моделей

## 3. Архитектура

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  ML-команды     │────▶│   REST API       │────▶│   SQLite        │
│  (train scripts)│     │   (FastAPI)      │     │   (метаданные)  │
└─────────────────┘     └────────┬─────────┘     └─────────────────┘
                                 │
                                 ▼
                        ┌──────────────────┐
                        │   Файловое       │
                        │   хранилище      │
                        │   (models/)      │
                        └──────────────────┘
```

**Компоненты:**

| Компонент | Назначение |
|-----------|------------|
| **REST API (FastAPI)** | Единая точка входа, валидация, логика |
| **SQLite** | Метаданные, поиск, версии — без отдельного сервера БД |
| **Файловая система** | Артефакты моделей — просто и предсказуемо |

**Почему так:**
- SQLite — один файл, бэкапы и перенос простые
- FastAPI — асинхронность, автодокументация, типизация
- Файлы на диске — совместимо с текущим workflow, можно мигрировать постепенно

## 4. API и схема БД

### Схема БД

```sql
-- Модели (логическая сущность)
CREATE TABLE models (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    team TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Версии модели
CREATE TABLE model_versions (
    id TEXT PRIMARY KEY,
    model_id TEXT NOT NULL REFERENCES models(id),
    version INTEGER NOT NULL,
    path TEXT NOT NULL,           -- путь к артефакту
    status TEXT DEFAULT 'staging', -- staging | production | archived
    metadata JSON,                -- произвольные поля
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(model_id, version)
);

-- Теги для поиска
CREATE TABLE tags (
    model_version_id TEXT REFERENCES model_versions(id),
    key TEXT,
    value TEXT,
    PRIMARY KEY (model_version_id, key)
);
```

### API (основные эндпоинты)

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/models` | Зарегистрировать модель + загрузить артефакт |
| GET | `/models` | Список моделей (фильтры: team, name, tag) |
| GET | `/models/{id}` | Детали модели и её версий |
| GET | `/models/{id}/versions/{version}` | Скачать артефакт |
| PATCH | `/models/{id}/versions/{version}` | Обновить статус/метаданные |
